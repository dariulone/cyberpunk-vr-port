User decision (2026-06-10): native stereo **на уровне D3D12** — единственный оставшийся путь к настоящему одновременному стерео (все engine-level рычаги мертвы [[cp2077-native-simultaneous-exhausted]]; Skip Tick опровергнут эмпирически — гостинг AER от head-pose mismatch, не от движения мира; UEVR Skip Draw требует IStereoRendering, которого нет).

**Архитектурный ключ (почему здесь может жить то, что умерло на engine-level):** engine-level replay падал на CPU-side re-RECORDING (неидемпотентная запись команд, [[cp2077-inline-replay-dead-all-levels]]). На D3D12-уровне повторяется только GPU-исполнение уже записанных закрытых command-list'ов — D3D12 явно разрешает multi-submit закрытого листа. Модель «in-place»: НЕ клонировать 35–40 RT ([[cp2077-d3d12-replay-feasibility]]), а перезаписывать те же промежуточные RT вторым проходом (как сам движок делает в следующем кадре).

**M0 probe (BUILT+DEPLOYED 2026-06-10, dxgi.dll md5 77a26fdf):** в `HookedExecuteCommandLists` (dxgi_factory_wrapper.cpp) первые K батчей present-очереди каждого кадра сабмитятся ВТОРОЙ раз сразу после оригинала (back-to-back — окружение данных идентично). Защита от Reset: фрейм-fence самого движка ставится в очередь ПОСЛЕ наших дубликатов => движок не сбросит листы/аллокаторы, пока дубликаты не исполнятся. Дубликаты НЕ перезаписывают камеру (M0 = тест стабильности/стоимости, картинка должна быть идентична).

**RUN (user):** запустить игру, в мире нажать **F3** (лог: `[STEREO-M0] double-submit ARMED`), окно K по умолчанию 6; **PgUp/PgDn** = K ±4. Каждые ~2 c при ARMED лог `[STEREO-M0] armed=1 window=K batches/frame=N doubled/frame=D frameMs=…`. Растить K через PgUp до batches/frame (= весь кадр), смотреть на стабильность и frameMs. Прислать `bin\x64\cyberpunkvrport.log`.

**DECODE:** (1) `batches/frame` = размер кадра в батчах (первая метрика, логируется и без ARM каждые ~15 c). (2) Рост `frameMs` при росте K = реальная цена simultaneous (ожидание ~×1.9 на полном окне). (3) DEVICE_REMOVED => DRED breadcrumbs уже подключены — лог покажет, какой лист/оп умер (=> какие батчи исключать/компенсировать барьерами). (4) Визуальная порча при идентичной камере = неидемпотентные батчи (UAV-append/счётчики) => кандидаты на исключение. Успех M0 = стабильность на большом K => зелёный свет M1.

**M1 (после M0): patched replay = настоящий правый глаз.** Записывать (root-CBV VA) сабмиты сценических батчей; найти view/proj-производные матрицы в upload-heap'ах ПО ИЗВЕСТНЫМ ЗНАЧЕНИЯМ (точные матрицы кадра известны из engine-side камеры [[redengine-render-view-state]] / DLSS-хука) — байтовый поиск, не статистика; после fence «левый глаз готов» патчить эти CB на правый глаз и ре-сабмитить сценическое окно; финальный scene-color копировать per-eye; culling FOV расширить (гейм-камера уже под контролем). UI у правого глаза будет отсутствовать => композитить UI-RT самим или оверлеем.

**Археология прошлой CBV-фазы (ускоряет M1):** tools/Launch-Cyberpunk-CBV-*.bat + Analyze-CbvTsv.ps1 + grouped_analysis.txt — существовал env-var-управляемый экспериментальный d3d12-прокси (из дерева удалён, переменные `CYBERPUNKVR_PATCH_CAMERA_CBV*` никем не потребляются). Результаты: per-draw CBV статистика — самый горячий биндинг draw root=2 entry=0 size=256 (36 меняющихся float, гигантские диапазоны = мировые позиции), чистой 4×4 проекции НЕ найдено => подтверждает декомпозированное хранение; конкретные пробы root18-entry7 / root19-entry1 / FLOAT=34 BIAS=0.5 — кандидаты прошлых сессий, статус неподтверждён.

**Стоимость/качество:** просадка ≈ как AER на половинном Present-rate (та же суммарная GPU-работа: 2 полных рендера за Present вместо 1 за каждый), но БЕЗ гостинга. DLSS/RT: replay делит единственную NGX/denoiser-историю => v1 с DLSS/TAA off для правого глаза или общая история с артефактами; per-eye история = поздняя фаза ([[cp2077-dlss-rt-vs-stereo-approach]]).

## RUN#2 (2026-06-10) — self-exclusion НЕ спас; диагноз = барьеры; M0.5 деплоен

**RUN#2 (md5 26502d2c, наши Present-сабмиты исключены):** window=6 ИГРОВЫХ батчей жил ~43 кадра (doubled/frame=6, totalDoubled=258), синтетический PgUp до 7 => лог замолк мгновенно, игра повисла, через 120 c движковый watchdog убил процесс (краш-репорт: Assert `Watchdog timeout! (120 seconds)`, engineWatchdog.cpp:198; DRED/device-removed НЕТ = GPU-очередь встала). Краш-репорты в `%LOCALAPPDATA%\REDEngine\ReportQueue\`.

**Синтез RUN#1+#2 (правка выводов RUN#1):** «жило на 16» в RUN#1 было везением короткого окна (~40 кадров); вис ловится на ЛЮБОМ окне ≥6 с вариативной задержкой 0.5–10 c. Теории «яд только в хвосте» и «яд в наших XR-копиях» ОПРОВЕРГНУТЫ. Рабочий диагноз: **mismatch resource-state барьеров** — батч оставляет ресурсы в состояниях ≠ начальным, дубликат гонит переходы с неверным before-state (на текстурах с метаданными depth/DCC это вис GPU).

**M0.5 (BUILT+DEPLOYED md5 df38090d): компенсация барьеров.** (1) Хук `ID3D12GraphicsCommandList::Reset` (slot 10, общий vtable direct-листов) + расширение хука `ResourceBarrier` (slot 26): при ARMED пишем все transition-барьеры каждого листа (skip BEGIN_ONLY half; ALIASING помечает лист опасным). Лист «tracked» только если его Reset видели ПРИ ARMED => первый armed-кадр всё скипает, со 2-го кадра дублирование. (2) Перед дубликатом: net-изменения по (resource,subresource) в порядке сабмита; компенсирующий лист (пул ≤64, fence-recycle, на устройстве очереди) переводит каждый изменённый ресурс end-state→start-state; затем повторный сабмит батча. (3) Скипаем батчи: untracked-лист / aliasing / смешанная гранулярность (ALL_SUBRESOURCES + конкретные sub на одном ресурсе). (4) Лог: `compBarriers/frame`, `skippedTotal`. Известные дыры (принято для зонда): implicit promotion/decay не компенсируется; UAV-данные неидемпотентных пассов могут портиться (артефакты допустимы — M0 меряет выживаемость, не картинку).

**Протокол RUN#3:** юзер запускает игру в мир, я через `tools\m0_step.ps1` (`-Key F3|PGUP|PGDN -Count N -WaitSec S`) армлю и поднимаю окно +1 с проверкой process.Responding + лога после каждого шага. ## M2 (BUILT+DEPLOYED md5 fadbdf68) — end-of-frame сериализованный full-replay

**Идея (обходит коллизию M0 таймингом):** M0 дублировал ВО ВРЕМЯ кадра (дубль гонится с async, читающим transient) — вис. Камера та же (тест ВЫЖИВАНИЯ, не картинки). M0-дубль в M2-режиме выключен.

**Механика:** flags bit4 (0x10) в файле маски. Захват в ECL-хуке (present+!ourWork+armed+0x10): AddRef листов в g_m2Captured, return (без M0-дубля). `M2Replay()` после originalFn(Present) в HookedPresent: swap g_m2Captured, реал-ECL (мимо хука) каждый батч в порядке + fence-маркер на батч (watchdog ловит вис), Release. Лог `[STEREO-M2] replayBatches/frame replayLists/frame totalReplays`.

**Протокол M2-run:** запуск в мир → F3 (arm) → лог `[STEREO-M2] mode=1 replayBatches/frame=~14`. ВЫЖИЛ (минуты, картинка норм/с артефактами) => сериализованный end-of-frame replay побеждает => copy-вариант жив; ДАЛЕЕ: swap camera-CBV (overwrite контента CB — VA запечён, контент читается на исполнении) перед реплеем + copy финального scene-color в правоглазый буфер => настоящее sequential-стерео. - **Паттерн класса фидеров:** numLists=1, мало переходов, БЕЗ indirect, позиция СРАЗУ ПОСЛЕ тяжёлого сегмента (4 после aliasing-пары 3/5-зоны, 7 после сценового 6). Это async-кики между сегментами кадра: present-фидер пишет вход + Signal, async-очередь обрабатывает, present.Wait перед следующим фидером. Кандидаты в тот же класс: 10, 13 (1 лист, мало trans) — проверить отдельно.
- **Маска засеяна 0x2407** {0,1,2,10,13} на следующий запуск: чисто => 7 виноват + 10/13 невиновны; вис на ord10/13-сигнатуре => и они фидеры.
- m0_step.ps1: удержание клавиши 80→400 мс (короткие тапы терялись при хитчах present-потока в начале сессии; 2 пропущенных F3 подряд).
