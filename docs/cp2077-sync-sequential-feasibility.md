User goal: both eyes from the SAME sim/world state → no inter-eye ghosting, full quality, DLSS+RT working. Question: will Sync Sequential deliver that, and can we build it from current data?

**WILL IT DELIVER (honest):**
- Eliminates inter-eye ghosting: YES — both eyes sample one frozen sim state = a correct stereo pair (the "фигня" = L/R from sim P vs P+1 is exactly what this fixes).
- DLSS + RT: run NATIVELY per frame (each eye is a full native engine frame) — same as RealVR ships. Caveat: alternate-eye shares the engine's single DLSS/RT temporal history bouncing L/R by IPD; motion vectors partially handle it; acceptable in practice (RealVR proof), not perfect per-eye history. ([[cp2077-dlss-rt-vs-stereo-approach]])
- COST (unavoidable): world sim/animation runs at HALF rate (≈45Hz @ 90fps display) because sim advances once per eye-pair. Head tracking stays smooth (compositor reprojects per-eye). GPU cost = SAME as current AER (still 1 view/frame) — no extra rendering. It is NOT full-rate genuine-simultaneous (that needs 2 views/frame = native, DEAD); it's "consistent pairs at half world-rate."
- vs AER V2: Sync Seq = honest 45Hz, clean, no synthesis artifacts, simple. AER V2 = target 90Hz via optical-flow synthesis, but synthesis artifacts. Different smoothness/quality tradeoff.

**CAN WE BUILD FROM CURRENT DATA (code-grounded):**
- ALREADY DONE (~80%): AER pair submit, pair-id cadence, per-eye camera shift (OnPatchCameraCallback), synced-pose latch. `openxr_manager.cpp` latchSyncedSequentialPair (~3081) + syncSequential path (~3069-3164).
- CONFIRMED GAP: current `xr_sync_sequential` is POSE-ONLY — latchSyncedSequentialPair touches ONLY m_synced* poses/views, NEVER engine sim/time. So the engine STILL advances sim between the two eye renders → the two eyes are STILL different sim states → ghosting NOT actually fixed. The flag is a cosmetic pose-lock (matches the old "cosmetic stub" note).
- MISSING 20% (the real work): freeze the engine's per-frame sim/time advance on alternate frames so eye-1 reuses eye-0's world state. This function is NOT yet identified.

**HOW TO BUILD THE MISSING PIECE (concrete, tractable):**
- CP2077 HAS a native "freeze world, keep rendering camera" primitive: Photo Mode (time scale → 0, free camera), plus pause + Sandevistan time-dilation. Don't invent a freeze — piggyback the engine's own time-dilation/delta-time variable (the one Photo Mode/pause sets to 0), toggling it per alternate frame.
- Starting clue already in code: MenuMode hook (OnMenuModeHookCallback, dxgi_proxy.cpp ~2867) — menu pauses the world; trace what it gates. Find the global engineTime/sim delta multiplier; force 0 on the "freeze" (eye-1) frame, restore on eye-0.
- This is a FOCUSED single-gate RE task, NOT a wall like multi-view registration ([[cp2077-native-simultaneous-exhausted]]).

**RISKS of the freeze (validate at test):** rapid 45Hz toggling of time-scale is untested (Photo Mode freezes for long spans, not oscillating); audio/async/input systems may not respect a per-frame frozen delta → possible micro-hitches; DLSS motion vectors with frozen world + IPD-jumping camera need a look.

**VERDICT:** Sync Sequential is the most realistic path to the user's actual priorities (no ghosting + DLSS + RT + same perf as now). ~80% built; the missing piece (engine sim-time freeze, ideally via native time-dilation piggyback) is tractable, unlike every dead native-simultaneous lever. NEXT STEP: reverse the engine's per-frame sim/time-advance (or time-scale var) to gate it on alternate frames. Relates [[redengine-job-render-arch]] [[aer-render-pose-submit]].

**USER Q: doesn't UEVR do engine-level Skip Tick (freeze EVERYTHING)?** Honest answer: yes — UEVR Skip Tick skips the whole UWorld tick, a more COMPLETE freeze. Our dilation is a SOFTER freeze (game-time only), leaving real-time-clocked visuals running. BUT for GHOSTING it's equivalent: the residuals (flat banners on static surfaces, screen-space HUD markers, static-position lights) don't move in 3D between the two eye views → they don't create stereo ghosting. The things that DO ghost (NPCs/vehicles/V/physics) ARE frozen. A true CP2077 engine-tick-skip = the same job-graph reverse wall (game-update job is pointer-dispatched, statically unreachable) + risks breaking input/audio/render-handoff (the tick does more than sim). So dilation = pragmatic, working, VR-safe equivalent for the no-ghosting goal; true tick-skip buys cosmetic completeness for high cost/risk.
