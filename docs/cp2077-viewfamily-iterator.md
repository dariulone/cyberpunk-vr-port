Found the producer-side view-family iterator and per-view scheduler (the True-Stereo right-eye injection point). Chain, outer→inner:

- **sub_140952B30** (RVA 0x952B30) — calls the iterator TWICE (+0x81, +0xAF) = two view groups/passes.
- **sub_140952C28** (RVA 0x952C28) — THE VIEW-FAMILY ITERATOR. Loops over a collection: outer over 16-byte group descriptors `a1[a4..a5]` (a4=start,a5=end index); inner over **128-byte view elements** (`v10 += 128`, end = start + count(@+12)<<7). For each element with active flag `*(BYTE*)(v10+120)` it calls the scheduler with spec=&{v10, fields from a3} and context `*(QWORD*)(v10+112)`.
- **sub_140952CF4** (RVA 0x952CF4) — PER-VIEW TASK SCHEDULER. Builds a job descriptor {entry=sub_1402437B0, payload=alloc(32), context=a1}, copies the view spec a5 into the 32-byte payload (*a5, a5[2], flag@24), then enqueues via sub_14014299C(qword_143438980=job system, ...). One call = one render-view job.
- **sub_1402437B0** (RVA 0x2437B0) — the view-task job entry (job-scope wrapper → sub_140243980 → virtual vtable[5] on the view object → camera system sub_140126D4C → locate sub_140127F58). Runs on a redDispatcher worker (confirmed via x64dbg [[cp2077-view-injection-mirror]]).

**M1 DONE (hook stable):** sub_140952CF4 hooked stably via PatchCodeSafely [[viewsched-hook-crash]]. Per-frame ~40 view-STAGES scheduled (not 40 cameras). 128-byte element decoded: +0x00 stage const (0xC00 main / 0xC03 secondary), +0x08 ptr to scene-view OBJECT (camera/data behind further ptrs), +0x60 stage-vtable (discriminator), +0x70 ctx, +0x78 active flag. Main player view = viewIdx=0, object name at obj+0xA0 ASCII ("03_night_city" in-world / "04_main_menu" in menu), elem-vtable RVA 0x2C5ECC0. Many of the 40 elements are different STAGES of the SAME scene-view object.

**M3 RESULT — naive double-submit is DEAD (empirically proven):** re-calling sub_140952CF4 on the main view 1× then 8× per frame for 900 frames = ZERO extra GPU cost, FPS unchanged, no crash. Therefore **the view-task scheduler is NOT the render trigger — it is idempotent per-frame view SETUP.** GPU scene-render count is determined by the number of distinct view OBJECTS the downstream FrameGraph processes (reading the single slot renderer+0x4658 [[aer-render-pose-submit]] / [[redengine-render-view-state]]), NOT by how many tasks we schedule. Re-running setup just re-writes the same data to the same slot.

**Revised option-C path:** True Stereo requires a GENUINE second scene-view OBJECT (right-eye), exactly like the engine's mirror/reflection system creates one [[cp2077-view-injection-mirror]] — the mirror IS rendered as a real 2nd full scene pass (confirmed Nsight + GPU cost). Right-eye = a "pseudo-mirror" view object facing forward, IPD-shifted, rendering into the right-eye RT. Next investigation: find where scene-view OBJECTS (the 128-byte element collection / mirror view object) are CREATED & registered for the frame — that producer is the real injection point, not the task scheduler. See [[redengine-job-render-arch]], [[cp2077-pass-census]].

**CORRECTION (idx0 was misidentified):** the obj at elem+0x08 of idx0 has a Wwise/audio-flavored vtable (RVA 0x2B529A0: AK::MemoryMgr/WriteBytesCount stubs, 'Audio/WeaponAudioComponentUpdate' string adjacent) and its vtable[5] is an AK stub. So idx0 is an AUDIO task, NOT the render view -- which fully explains M3's zero FPS effect (we double-submitted audio, not a view). The '03_night_city' name at obj+0xA0 is a world-SECTOR tag shared by many component types, not a render-view marker.

**sub_140952CF4 is a GENERIC per-frame TASK scheduler, not the render-view family.** Probed all 48 scheduled elements live: each has a DISTINCT object vtable + DISTINCT vtable[0] destructor (48 different classes: audio, anim, streaming, ...), but ALL share **vtable[5] = sub_14014A700** -- the common 'execute task' virtual the job (sub_1402437B0 -> sub_140243980) invokes. Render-view is ONE of these 48 tasks but is NOT distinguishable by vtable[5]; sub_14014A700 must dispatch by type internally. Next: decompile sub_14014A700 (RVA 0x14A700) to see the per-type dispatch and how to identify/inject the render-view task. Injecting a right-eye view now means constructing a valid render-view TASK object -- another layer deeper. See [[cp2077-viewfamily-layout]].

**RENDER-VIEW TASK IDENTIFIED (runtime correlation, definitive):** Hooked sub_140243980 (task dispatch, RVA 0x243980; prologue mov[rsp+10],rbx = 5-byte E9) to stash its a1 (=task struct, rcx) into a TLS slot per worker thread; read that TLS inside the existing locate hook (locate runs synchronously inside the render-view task). Result: the render-view task has **taskWork = sub_140D69018** (RVA 0xD69018), reached via task-struct whose vtable ptr is at +0x28 (taskVt RVA 0x3334780, taskWork = *(taskVt)[0]). Only ONE distinct taskWork reaches locate => main view + mirror share this task class. NOTE: this task struct is NOT one of the 48 elements from the sub_140952CF4 iterator (those had different addrs and their +0x28 was a different/garbage field) -- so the iterator/scheduler we probed earlier is a DIFFERENT (generic) subsystem, and the render view is dispatched via sub_140243980 with a distinct struct layout. Next: decompile sub_140D69018 (the render-view task body) to learn how the view/camera/RT is set up = the template for a right-eye clone. See [[cp2077-viewfamily-layout]].
