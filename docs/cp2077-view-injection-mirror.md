Runtime probe (MONO mode, standing at V's apartment mirror with reflections on) on the locate hook `OnLocateCameraCallback` (sub_140127F58 view-descriptor builder). Counted locate calls per present-frame:

- viewIdx=0 (main view): stable descriptor object `...800xx` (mostly `...80010`).
- viewIdx=1 (MULTI): the **mirror/reflection view** — a SEPARATE, transiently-allocated descriptor (`...81xx-8Axx`, new address each frame) whose quat tracks the reflection camera orientation as the player turns. Max viewIdx=1 (exactly 2 views/frame at a mirror; ~1/frame outdoors with no mirror).

**Breakthrough for option C (true simultaneous stereo):** the engine builds a per-frame SET of render-view descriptors through locate, and a full secondary scene-view (mirror, full pipeline per Nsight [[cp2077-pass-census]]) is one of them. So a right-eye view can be injected by adding another descriptor to this per-frame set — the "view family" that did NOT exist at the camera-blend level ([[redengine-render-view-state]] sub_14012752C was a temp blend array) DOES exist at the render-view-build level, and the mirror proves it's extensible/re-invokable. Earlier "buried in the job-graph, no accessible injection point" verdict ([[redengine-job-render-arch]]) was too pessimistic.

**Still open (the real C work):** (1) how a view descriptor maps to its render target (mirror renders to a small mirror RT — right-eye needs full-res eye buffer); (2) how to make the engine actually render an injected descriptor (vs just observe); (3) compositing the injected view as the right eye. Note: under STEREO/AER the mirror render breaks (flickers) — the mod's per-eye camera path corrupts secondary views; the probe only works in MONO.

**RIGOROUS x64dbg RESULT (reliable, .pdata unwind):** HW BP on locate (cyberpunk2077.exe+127F58) at the mirror caught a clean worker stack (redDispatcher14): locate(127F58) → 1275FB → 126D90 → 2439F9 → 243838 (render-view TASK body) → 14324C (job-loop sub_140143060) → 16D50E → 1A3ABF4 (job thread). The chain bottoms out in the JOB LOOP. So main view and mirror view are the SAME task sub_140243838 pulled from the job queue with different data — the worker stack is identical for both, and does NOT reveal who enqueued the view (producer/consumer are decoupled by the queue). The "view-family iterator" is producer-side (the enqueue of view-tasks), on another thread behind the queue — NOT reachable from the worker stack. CONCLUSION: option C's right-eye injection point is buried on the producer side of the job system; reaching it needs a separate hunt for the view-task enqueue site + building a valid task + RT routing + compositing = the expensive multi-week path for ~12%. Two concrete C paths remain, both costly: (a) find+hijack the view-task enqueue (producer side); (b) hook the task body sub_140243838 and double-invoke it for the right eye (re-entrancy risk). Recommended instead: Synchronized Sequential (pre-authorized fallback).

**FALSE LEAD (do not re-chase):** raw-stack-scan on the mirror-view locate call surfaced 0x14057D690 / 0x14057D56A / 0x14057D4F4 — these are NOT the view dispatcher. They are the generic render-proxy update-command queue (RTTI strings nearby: `UpdateMaterialParameterEffectContext`, `UpdateVehicleParams`, `DisableClusteredProxyInstances`); the 192-byte element is a material-param update command. They were STALE stack values picked up by the unreliable raw scan, not the live caller of locate. The reliable mirror-view chain is the SAME as the main view (127E66 → 126D90 → 243838 → ... → job 1A3ABF4) — the view-family iteration is at/above that level. Raw stack scanning is too unreliable to pin it (gave a false positive); pinning the view iterator needs a proper debugger (HW breakpoint on locate + real unwind), not the scan.
